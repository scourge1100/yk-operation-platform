# Technical Decisions

본 문서는 시스템에서 사용된 주요 아키텍처 및 기술 선택과 그 이유에 대해 설명합니다.

초기 아키텍처는 선행 설계되어 있었으나,
백엔드 기능을 구현하는 과정에서 해당 구조를 분석하고 이해하며 개발을 진행했습니다.

---

## Interface Server 기반 구조

본 시스템은 Admin 플랫폼과 게임 서버 간 통신을 위해 Interface Server를 사용합니다.

게임 서버는 보안상의 이유로 외부에서 직접 접근이 불가능하기 때문에,
모든 운영 요청은 Interface Server를 통해 전달됩니다.

이 구조를 통해 다음과 같은 효과를 얻을 수 있습니다:

* 외부 요청을 단일 진입점에서 통제
* 요청 검증 및 처리 흐름 중앙화
* 내부 시스템 보호 및 보안성 향상

해당 구조를 이해하는 것은 외부 시스템과의 안정적인 연동 기능을 구현하는 데 중요한 요소였습니다.

---

## Bastion Server 기반 보안 통신

게임 서버는 내부 네트워크에 위치하며,
외부와의 통신은 Bastion Server를 통한 SSH 터널링 방식으로 이루어집니다.

이 설계를 통해:

* 내부 서버에 대한 직접 접근 차단
* 통신 경로 통제
* 민감한 운영 요청의 안전한 처리

가 가능하도록 구성되어 있습니다.

---

## 데이터 접근 구조 (Interface 기반 접근)

운영 플랫폼은 MariaDB를 직접 사용하여 운영 데이터를 관리하며,
게임 데이터(MSSQL)는 직접 접근하지 않고 Interface Server를 통해 처리됩니다.

이 구조는 다음과 같은 특징을 가집니다:

* 운영 시스템은 자체 DB(MariaDB)만 직접 사용
* 게임 데이터는 Interface Server를 통해 간접적으로 접근
* 데이터 접근 경로를 명확히 분리

이를 통해 다음과 같은 효과를 얻을 수 있습니다:

* 게임 데이터베이스에 대한 직접 접근 차단
* 시스템 간 결합도 감소
* 보안성 향상 및 데이터 무결성 유지

기능 구현 시, DB 조회가 아닌 API 호출을 통해 데이터를 처리하는 구조를 이해하는 것이 중요했습니다.

---

## 계층형 아키텍처 (Controller / Service / DAO)

백엔드는 계층형 아키텍처 구조를 기반으로 구성되어 있습니다.

* Controller: 요청 처리
* Service: 비즈니스 로직 수행
* DAO: 데이터베이스 접근

이 구조는 다음과 같은 이점을 제공합니다:

* 유지보수 용이성
* 관심사의 분리
* 코드 재사용성 향상

기능 구현 시 일관된 구조를 유지하기 위해 해당 계층 구조를 기준으로 개발을 진행했습니다.

---

## 서버 사이드 렌더링 (JSP)

본 시스템은 JSP 기반의 서버 사이드 렌더링 방식을 사용합니다.

이 방식은 다음과 같은 특징을 가집니다:

* 백엔드와의 통합이 용이
* 레거시 환경과의 호환성 확보
* 별도의 프론트엔드 인프라 없이 개발 가능

다만, UI와 비즈니스 로직 간 결합도가 높아질 수 있어 주의가 필요합니다.

---

비록 초기 설계는 선행되어 있었지만,
구조를 분석하고 이해하는 과정을 통해 안정적인 기능 구현이 가능했습니다.

이 경험을 통해 시스템 아키텍처를 이해하고,
그 위에서 일관성 있는 개발을 수행하는 역량을 키울 수 있었습니다.
